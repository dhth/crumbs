# Storage

The tool uses a local SQLite database for persistent storage.

This is intended to support:
- local-only usage
- persistence across CLI and TUI restarts
- multiple agent sessions writing progress concurrently
- a simple relational model for sessions and crumb history

## Core Data Model

The initial storage model uses two relational tables:
- `sessions`
- `crumbs`

`crumbs` rows are linked to `sessions` through a foreign key.

## Sessions Table

The `sessions` table holds durable identity, current state, and session
metadata.

Expected fields include:
- `id`
- `agent_name`
- `title`
- `project_name`
- `path`
- `branch` nullable
- `state`
- `archived_at` nullable
- `created_at`
- `updated_at`

Notes:
- `session_id` is the primary stable identity for a session.
- It is stored as an integer and can map directly to the SQLite primary key.
- `updated_at` reflects the timestamp of the latest accepted crumb.
- `archived_at` is nullable and supports soft archiving for sessions.
- default session queries return only active sessions where `archived_at` is
  `NULL`.
- `branch` is nullable because not every working directory is on a git
  branch.

## Crumbs Table

The `crumbs` table holds the append-only progress history for each
session.

Expected fields include:
- crumb primary key
- `session_id` foreign key
- `message`
- optional `state`
- optional `confidence`
- `created_at`

Notes:
- each `crumbs` row represents one accepted `crumbs log ...` event
- crumb timestamps are generated by the tool, not supplied by agents
- crumbs remain append-only
